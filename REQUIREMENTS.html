项目目标（iScience 投稿导向）

构建一个多 Agent 协作的视疲劳诊断系统，核心创新点：

  • 将训练好的视觉模型作为 tool 暴露给 LLM agent
  • Agent 综合行为指标 + 患者主诉 + RAG 知识库 + 历史相似病例 → 给出可解释的诊断推理
  • 病例库随使用增长，反哺新病例的诊断（"系统越用越聪明"）
  • 所有诊断推理可追溯、可消融实验，满足 iScience 对方法学透明度的要求

二、技术栈（已决定，不要替换）

  • Agent 编排：LangGraph（状态机式，符合诊断流程）
  • LLM 后端：Ollama（通过 langchain-ollama 接入）
  • 视觉模型：作为 FastAPI REST 服务独立运行，agent 通过 HTTP 调用
  • 病例库：PostgreSQL + pgvector（结构化字段 + 向量字段）
  • 嵌入模型：BGE-M3（中文医学文本检索效果好，通过 sentence-transformers 本地运行）
  • 知识库 RAG：LlamaIndex
  • 配置管理：pydantic-settings，所有参数走 .env
  • 日志：loguru，所有 agent 决策必须可追溯

三、系统架构

请实现以下 LangGraph 状态图：

[START]
  → IntakeAgent       (采集患者基本信息、主诉、量表分数 CVS-Q/OSDI)
  → VisionAgent       (调用 FastAPI 视觉服务，获取行为指标)
  → FeatureAggregator (时序统计 + 异常值检测 + 指标归一化)
  → RetrievalAgent    (RAG 检索: 指南文献 + 相似历史病例)
  → DiagnosisAgent    (Chain-of-thought 推理，输出诊断 + 严重度分级)
  → ConsistencyCheck  (Self-consistency: 独立推理 3 次取一致结果)
  → CaseLibraryAgent  (结构化入库 PostgreSQL)
  → ReportAgent       (生成医生版 + 患者版双报告)
[END]
每个节点的 State 必须包含上游所有节点的输出，便于失败时回溯和消融实验。

五、关键设计约束

  1. 每个 agent 必须可独立调用和单独评估 —— 这是消融实验的硬性要求。 每个 agent 都暴露 run(state) -> state 接口， 不要内部 hard-code 调用其他 agent。
  2. 所有 LLM 调用必须记录：input prompt、output、token 数、耗时、模型版本。 存到 logs/ 下的 JSONL，每个患者一个文件， 文件名 {patient_id}_{timestamp}.jsonl。
  3. 视觉模型集成方式：先实现 services/vision_api.py 的 FastAPI 框架 （端点 /analyze，接收视频路径，返回行为指标 JSON）， 留出 TODO 让我把现有的 Detectron2 推理代码贴进去， 不要尝试生成 Detectron2 推理代码。
  4. 病例库 schema 至少包含：patient_id、采集时间、主诉文本、 量表分数（结构化字段）、行为指标（JSONB）、诊断结论、严重度分级、 agent 推理链（text）、医生复核结果（可空，用于后续监督）、 视频文件路径、症状文本的 embedding 向量（用于相似病例检索）。
  5. Self-consistency 实现：DiagnosisAgent 用不同 temperature （0.3 / 0.5 / 0.7）跑 3 次，ConsistencyCheck 节点用多数投票 + 置信度分析； 如果三次结果分歧大（如严重度跨两级），State 标记 requires_human_review=True。
  6. 配置驱动消融：在 config 里加 ablation_mode 字段， 可以一键关闭某个 agent（如 disable_rag=True、 disable_consistency=True），便于实验。
  7. 不要过度工程化：第一版不需要 web UI、不需要 Docker、不需要异步并发。 CLI 跑通端到端流程即可。
