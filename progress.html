三、已知 Bug：RAG / 相似检索死锁

现象：demo 进程跑到 retrieval 节点 → 调 rag_retriever.retrieve_guideline() 或 embed_text() 后整个进程挂在 futex_wait_queue_me，CPU 几乎不动，GPU 0 %，11+ 分钟无 LLM 调用产生。强制 kill 才能恢复。

已排除的可能

  • ✗ BGE-M3 模型自身：单独进程 4 秒加载 + 0.46s embed，正常。
  • ✗ HuggingFace 网络：HF_HUB_OFFLINE=1 后仍死锁。
  • ✗ 全局 SOCKS 代理：NO_PROXY=localhost,127.0.0.1 已设，且 BGE-M3 单跑能成功。
  • ✗ Ollama 异常：intake LLM 调用已成功完成，证明 ollama 路径正常。

可能的根因（待验证）

  1. LlamaIndex × torch fork 死锁：demo 进程已经 import 了 langgraph / langchain-ollama / httpx，可能给 BG worker 留下了某种锁状态，第二次（在 retrieval 节点）调 sentence-transformers/torch 时与 tokenizers 的 fork 守卫冲突。
  2. LIDSettings.llm = None 触发 llama-index 内部 MockLLM 初始化时与 BGE-M3 的 CUDA 上下文初始化竞态。
  3. 多线程 lazy lock：rag_retriever._get_embed_model 的 threading.Lock 与上游 LangGraph 的某个 thread/asyncio 状态死锁。

排查计划（按优先级）

  1. 在 graph.compile 之前预热 BGE-M3（强制在主进程早期完成 CUDA 初始化），观察是否解锁。
  2. 把 rag_retriever.retrieve_guideline 临时替换为只用 sentence-transformers 直调 + 简易余弦检索，绕开 LlamaIndex 框架，看 BGE-M3 在 demo 进程里能否完成 embed。
  3. 用 py-spy dump --pid X 抓死锁栈帧，定位卡在 torch / llama-index 的具体调用。
  4. 若仍无解，把 RAG 拆到独立子进程（类似 scripts/run_single_case.py 的隔离思路），通过 IPC / FastAPI 让 demo 进程只发 HTTP 调用，避免在主进程同时跑两个 PyTorch 子系统。
