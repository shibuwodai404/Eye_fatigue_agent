一、Motivation：为什么做这件事

资产 × 痛点 × 目标

已有资产单独看的问题组合起来的机会
眼部行为视觉模型
Detectron2 分割
输入眼部视频，输出眨眼频率 / 完整度、眼动幅度、注视稳定性等		 只产出 Excel + HTML 可视化，不下结论，临床场景仍要医生看图说话 把视觉模型作为 tool 暴露给 LLM agent，agent 综合 行为指标 + 主诉 + 量表（CVS-Q/OSDI）+ 指南 RAG + 历史相似病例 给出可解释的诊断，并把每次诊断回写入病例库—— 系统越用越聪明，且全过程在本地，数据不出域。
本地 LLM
Ollama
主机上已部署，GPU 可用		 裸模型不懂临床流程、不会调外部工具、推理结果一次性不可追溯
本地运行环境
GPU + 私有数据		 医疗数据有合规约束，不能走云端；但本地缺现成多 agent 编排框架

iScience 投稿的方法学卖点

① Tool-augmented Diagnosis

视觉模型作为可调用的 FastAPI 服务，LLM 通过结构化指标推理，避免"端到端黑盒"。

② Self-improving Case Library

每例诊断结构化入库（PG + pgvector），后续病例自动检索相似先例反哺推理。

③ Transparent & Ablatable

每个 agent 独立可关、每次 LLM 调用全量 trace，方法学透明度直接服务消融实验。

二、核心功能：系统能做什么

一次问诊的端到端输入 / 输出

输入系统处理输出
• 患者眼部视频 (mp4)
• 主诉自由文本
• CVS-Q / OSDI 量表总分
• 人口学信息 (年龄 / 性别 / 屏幕时长) 		8 节点 LangGraph 顺序流水线（详见第三节），其中 诊断节点用 3 个 temperature 并行采样， 一致性节点做多数投票 + 置信度评估 • 诊断标签 + 严重度（none/mild/moderate/severe）
• 置信度 + 投票分布
• 医生版报告（含推理链与证据引用）
• 患者版报告（通俗语言）
• 是否触发 requires_human_review
• 完整 state JSON + LLM trace JSONL

关键运行能力

  • 可消融 — 通过 .env 一键关闭：DISABLE_RAG / DISABLE_CONSISTENCY / DISABLE_CASE_LIBRARY / DISABLE_SIMILAR_CASES。
  • 可追溯 — 每次 LLM 调用记录到 logs/llm_traces/{patient_id}_{session_id}.jsonl，包含 prompt / output / tokens / latency / temperature。
  • 降级 friendly — 病例库默认 JSONL fallback；装好 PG + pgvector 后改 CASE_DB_BACKEND=POSTGRES 无缝切换。
  • 视觉模型解耦 — Detectron2 推理代码留 TODO 占位，作为独立 FastAPI 服务运行（默认端口 8765），agent 通过 HTTP 调用。

三、框架思路：怎么搭起来的

编排层：8 节点 LangGraph 流水线

IO / 入库 工具调用 LLM 推理 聚合 / 决策
① IO
IntakeAgent
主诉文本 + 量表分 → LLM 抽取结构化字段（症状、风险因素等）
② TOOL
VisionAgent
HTTP 调用 FastAPI 视觉服务，拉取眨眼 / 眼动等行为指标
③ AGG
FeatureAggregator
归一化 (z-score) + 异常检测 + 综合疲劳分
④ TOOL
RetrievalAgent
RAG：LlamaIndex + BGE-M3 检索指南文献 + pgvector 找相似历史病例
⑤ LLM ×3
DiagnosisAgent
CoT 推理，3 个 temperature (0.3/0.5/0.7) 并行采样 3 份候选诊断
⑥ DECIDE
ConsistencyCheck
多数投票 + 置信度；分歧跨级 → requires_human_review=True
⑦ IO
CaseLibraryAgent
结构化字段 + 文本 embedding 入库 PG（或 JSONL 兜底）
⑧ LLM
ReportAgent
医生版 (含推理 / 证据) + 患者版 (通俗语言) Markdown 报告
图本身 不做条件分支 —— "禁用某节点" 已在 agent 内部用 settings.ablation.* 短路成 no-op。 这样消融实验只改 .env，图拓扑零变更。

四、State Schema 与数据流

State 是一个 TypedDict，累加式——每个节点只往里加自己的字段，不删上游产物。这让任意节点失败都能回溯，也让消融实验前后对照变得平凡。

GraphState
├── 元信息           : patient_id, session_id, timestamp, video_path
├── intake           : 主诉 / 量表 / 人口学 + LLM 抽取的 structured 字段
├── vision_metrics   : blink_rate, gaze_amplitude, fixation_stability, ...
├── aggregated       : normalized / anomalies / summary_stats / composite_score
├── retrieval        : { "guideline": [...], "case": [...] }
├── diagnosis_candidates : [ {temperature, label, severity, CoT, evidence}, ×3 ]
├── final_diagnosis  : label / severity / confidence / vote_distribution
├── requires_human_review : bool
├── case_library_record_id : str | None
├── reports          : { doctor_md, patient_md }
├── ablation_flags   : 本次运行启用的消融开关
├── llm_calls        : [ {agent, model, prompt, output, tokens, latency, ...}, ... ]
└── errors           : { agent_name: error_str }   # 单点错误不中断后续节点

五、关键设计取舍

  1. 顺序图 + agent 内部短路，不是条件分支图。
    理由：消融实验改 .env 即可，无需重编图；图拓扑稳定有利于审稿复现。
  2. Self-consistency 用 temperature 多样化，而非多模型（0.3/0.5/0.7 各跑一次）。
    理由：本地 Ollama 只有一份模型权重；temperature 采样在工程上最低成本拿到分布。
  3. 视觉模型不内联，做成独立 FastAPI 服务。
    理由：Detectron2 依赖重 / 模型已训好；解耦后 agent 工程不被视觉依赖污染，也方便 GPU 资源单独排布。
  4. 病例库双 backend（JSONL ⇄ PG + pgvector）。
    理由：第一版无需运维门槛；规模化后无缝升级，schema 已在 scripts/init_db.sql 落定。
  5. LLM 强制 JSON 模式 + 宽松解析兜底（llm.parse_json_loose）。
    理由：Ollama 偶尔返回带噪 JSON；截取首尾大括号兜底，避免单次抖动整链失败。
  6. 第一版不做 UI / Docker / 异步。
    理由：CLI 跑通端到端足够支撑论文实验；过早工程化会拖慢方法学迭代。

七、快速跑通

# 1) 准备环境（首次）
conda create -n eye-agent python=3.11 -y && conda activate eye-agent
pip install -r requirements.txt
cp .env.example .env

# 2) 确认 Ollama 在跑、模型已 pull
ollama list      # 期望看到 OLLAMA_MODEL 指定的那个

# 3) 启动视觉服务（先用 mock 实现，Detectron2 代码待贴入 TODO）
uvicorn src.services.vision_api:app --host 0.0.0.0 --port 8765

# 4) 跑端到端 demo
python -m scripts.demo_end_to_end --video /tmp/mock_video.mp4 --patient-id demo_001

# 5) 看产物
ls logs/reports/   # *.state.json / *.doctor.md / *.patient.md
ls logs/llm_traces/   # 每次 LLM 调用的全量 trace
